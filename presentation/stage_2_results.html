How It Works

Perplexity Agent API

API Parameters and how they drive cost

Parameter Current Value Description Tunability
model GPT-5.2 The underlying LLM that performs reasoning and generates the output. Can switch to Claude Opus 4.6 for potentially higher-quality reasoning (more expensive)
max_steps 10 Maximum number of search–reason iterations the model can perform. Each step adds all prior search results to the context window, so input tokens grow with each step. This is the primary cost driver. Range: 7–15. Lower = cheaper but may miss evidence. Higher = more thorough but significantly more expensive.
max_tokens 10,000 Maximum tokens in the model’s output response (our JSON findings). Range: 3K–15K. 10K is sufficient for our structured JSON output. Increasing wastes budget; decreasing risks truncation for companies with many findings.
max_tokens_per_page 4,000 How many tokens the model reads from each web page it fetches. Controls the depth of each individual source. Keep at 4K. Lower values cause the model to miss context on longer pages; higher values inflate cost per step without clear quality gains.
tools web_search, fetch_url The tools the model can invoke during reasoning. web_search queries the web; fetch_url reads full page content from a URL. Fixed for deep-research. Both are required for thorough evidence gathering.

Why max_steps dominates cost

Each reasoning step adds all previous search results to the context window. By step 7–10, the model is processing hundreds of thousands of accumulated input tokens. The model decides how many steps to take (up to the cap) — there is no way to predict in advance how many steps a given company will require.

In our tests, average input tokens ranged from 58K (Run 1) to 247K (Run 2) — a 4× difference driven entirely by the model taking more reasoning steps on the server side.

Key finding: cost is driven by Perplexity's server-side behavior

A/B test on identical companies: We re-ran the exact same 19 companies from Run 1 using the exact same prompt and parameters. The average cost was $0.60 instead of the original $0.16 — a 3.7× increase.

Input tokens jumped from ~58K to ~290K average, while search result counts remained the same. This confirms the model is performing more internal reasoning steps per call due to a server-side model update, not due to our code or prompt changes.

Implication for scaling

We cannot rely on a fixed cost estimate. The cost per company will fluctuate between $0.15 and $0.70 depending on Perplexity's model behavior on any given day. Our scaling methodology must account for this unpredictability.

Proposed Approach

Batch-Controlled Scaling

Rather than committing to a single large run, we process companies in monitored batches of 100. After each batch, we review the average cost and hit rate, then decide whether to continue, adjust parameters, or pause.

CLI Control Levers

The production script exposes several parameters that control cost and throughput:

Terminal production_agent_runner.py 
# Run 100 companies per priority, cap budget at $200
python -m src.stage_2.production_agent_runner \
    --sample-size 100 \
    --priorities 5 4 \
    --concurrency 10 \
    --budget-cap 200 \
    --max-steps 7
Flag Purpose Impact
--budget-cap Auto-stop when cumulative spend reaches this threshold Prevents runaway cost
--max-steps Override the preset's reasoning depth (default: 10) Fewer steps = lower cost, possibly lower hit rate
--sample-size How many companies per priority level Controls batch size
--concurrency Parallel API calls (Tier 1: 3 QPS, 150 RPM) Faster execution, same cost
Ctrl+C Pause (1st press), Stop (2nd press), Force exit (3rd press) Manual intervention at any time

Looking Ahead

Next Steps

A. Two-Pass Strategy for Higher Yield

Pass 1 classifies every "no finding" result into one of two categories via the no_finding_reason enum. This classification enables an intelligent second pass.

no_finding_reason enum
"limited_online_presence" Company has a weak digital footprint. Even a more powerful model will not find what does not exist online. → Skip in Pass 2 (save money) "has_presence_no_evidence" Company has an active website, blog, job postings, press coverage, etc. but no verifiable GenAI adoption evidence was found by the deep-research preset. → Re-process in Pass 2 with more powerful model

From our test data, ~90% of "no finding" companies are classified as has_presence_no_evidence. Pass 2 would re-process only these companies with the more powerful (and more expensive) advanced-deep-research preset (Claude Opus 4.6).

Optional enhancement: no_finding_analysis

We can add a free-text field where the agent writes 3–5 sentences explaining why no evidence was found. This helps us understand if the company is genuinely non-adopting vs. just hard to research, and informs whether Pass 2 is worthwhile. However, this field increases cost per call. We can enable it selectively.

B. Stage 3: Validation

All findings come with source URLs, but LLMs can hallucinate citations. Before including findings in the research paper, we need a validation layer. Two options:

Automated

Option 1: LLM-as-a-Judge

  • A separate LLM visits each source URL and checks if the finding is supported by the page content
  • Estimated cost: ~$20 for all findings
Human Review

Option 2: Validation GUI

  • Web interface showing finding + source URL side by side. Reviewer clicks Confirm or Flag.
  • Estimated time: 30–60 seconds per finding x ~2,000 findings
  • Or can validate a sample (e.g. 10%) and extrapolate precision
